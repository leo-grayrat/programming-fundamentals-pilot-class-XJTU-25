# 西安交通大学《程序设计基础》AI 试点班课内项目归档

>[!NOTE]
>本仓库**文字部分整理本身使用了 AI 辅助**，但源代码编写过程基本为主要自学手敲，并非 vibe coding 项目。
>关于文字整理 AI 辅助的详细介绍，参见“关于这些说明文档”部分。

## 这是什么

本仓库用于归档 2025-2026 学年西安交通大学《程序设计基础》（COMP300205，3 学分）课程“AI 试点班”的部分课内项目、原始报告书整理稿。**使用语言为 C 语言（C89标准）**，项目 1~3 基于 **Windows** 环境开发，项目 4 基于 **Linux** 云服务器环境开发。

仓库中的材料主要对应课程实践项目，包含一下四个项目：

- 贪吃蛇游戏
- 金融市场定价模拟
- 环境数据监测分析
- 矩阵乘法优化

这些项目最初大多以 `.docx` 形式撰写，并且需要遵循课程统一给定的报告书模板。因此，原始文档在体例和措辞上会明显偏向课程验收所要求的正式、板正风格，而不是面向真实课题组或工程团队的实验报告写法。

## 文件结构

- `original guide` 文件夹下是该项目原始指导书（已删除课程隐私信息）。
- `code` 内为源代码及可执行文件（若有）。
- `README.assets` 内为各项目子 `README` 文件内引用图片合集。
- `Release` 发布的是单独汇总的各项目（项目四除外）的可执行文件。

## 关于这些说明文档

需要说明的是，仓库中的许多说明文字并不是在项目完成当下就以 Markdown 形式直接写成的，而是由当年的课程报告、代码注释、排错记录与后续整理材料共同拼接而来。

换言之，这里公开的并不是“当时就已经写好的 GitHub 版本”，而是事后归档：

- 原始项目开发发生在 C 语言初学阶段；
- 原始文档必须服从课程模板，因此表述方式常常带有明显的作业化痕迹；
- 现在转为 Markdown 并整理到 GitHub，是站在后来的视角对旧项目做归档，而不是重新完整重写这些项目；
- 在这一轮整理中，**AI 参与了大量转写、补全、重组与表述修整工作**（包括这个说明文档的很大一部分...时间所限 orz）；
- 但受限于时间，整理者只能进行有限审核，无法对所有远古项目说明文档逐段返工。

展开来说，主要有两个方面的问题：

### **原始材料自身的局限**  

这批材料最初形成于 C 语言初学阶段，而且原始文档必须服从课程报告的体裁、格式和验收要求。也就是说，它们一开始就不是按公开仓库中的技术文档标准来写的，而是按课内提交物来组织的。  
因此，原文中难免会同时带有两类痕迹：

- 初学阶段的表达与判断方式；
- 课程报告常见的正式、板正、求稳的写法。  
这会使部分内容更强调“交代完整”“贴合模板”，而不是强调工程文档通常更重视的简洁、克制和信息密度。

### **后期 AI 再加工带来的偏差**  

这些材料在转写、补全和整理为 Markdown 的过程中，经过了 AI 的再加工；而这一步并不只是“润色”，也会引入新的问题。其核心问题主要集中在**身份错位**与**语义冗余**上。常见表现包括：

- **角色定位错位**  
  AI 往往难以稳定维持第一人称复盘口吻，容易跳到旁观者、审稿人或助手的姿态，对原作者的行为进行点评。  
  
  > “原报告中提到的这一处理逻辑，充分体现了作者当时扎实的基础。”

- **文风过度煽情**  
  AI 习惯性添加并无实质信息的感悟、升华或自我感动式总结，把原本只需平实记录的技术问题写得过满。  
  > “这次报错不仅是技术上的挫折，更让我深刻体会到了计算机世界严谨且迷人的魅力。”
  
- **信息边界模糊**  
  它容易把修改过程中的元叙述、对指令的响应或过程性解释直接缝进正文，导致内容明显出戏。   
  
  > “应你的要求，我已将库函数和系统环境部分进行了严格解耦，确保它们不再混淆。”

- **结构化思维僵化**  
  它常机械套用总分总、分维度分析或价值归纳，把本来只需要直接说明的问题写成套路化总结。  
  
  > “这一部分的实现，可以从学习痕迹、调试痕迹和设计痕迹三个维度来拆解其深层价值。”

- **这一轮整理的边界也需要说明。**  
  受限于时间，这次公开整理主要完成了：

  - 格式转换；
  - 内容归并；
  - 对 AI 生成部分明显问题的清理。  

  但并没有对所有旧文档逐段彻底重写。  
  因此文中若仍有个别表述不够自然，可从以上两个方面理解。

## 为什么公开这些材料

- **个人存档与备份**
  单纯是为了整理本地硬盘，把早期初学编程阶段写过的一些课程大作业统一搬到 GitHub 上做个长期备份，也算是在自己的学习轨迹上留个纪念。
- **提供一份可运行的参考样本**
  当年自己在做这类课程项目时，经常因为找不到完整的参考案例而踩坑。现在将它们开源出来，希望能给之后遇到类似需求、或者修读相关课程的同学提供一点具体的实现思路和“避坑参考”。
- **关于学术诚信的提醒**
  本仓库的所有代码与文档仅供学习交流和思路参考。**请务必不要直接复制粘贴并作为自己的课程作业提交**。若因直接抄袭导致学术违规或查重未过，后果请自行承担。

## 这批项目里能直接看到什么

> [!NOTE]
> AI 煽情警告！但是其总结的现象是真实存在的

- **能直接看到初学阶段留下的自我提醒。**  
  例如贪吃蛇代码里会直接写：
  
  - `// ==才是比较，不是=！！！`
  - `//for里面三个语句用分号隔开!!!`
  - `//有个很严重的问题，C89不支持for循环内部定义变量...`
  
  这种注释不是为了给别人讲概念，而是写代码的人在一边踩坑一边给自己留“别再错了”的提醒。它说明这些项目并不是事后重新誊写成的干净成品，而是把当时真实开发状态一起保留下来了。
  
- **能直接看到环境和语言标准带来的限制是怎么被记下来的。**  
  例如 `envdata.c` 一开头就写了：
  - `//要让整个项目统一使用ANSI (多字节字符集)`
  - `//我又要🐎C89了...`
  - `//但是C99才有...还是int吧孩子们`
  
  这里不是泛泛地说“注意兼容性”，而是把具体卡点直接写出来了：字符集要统一，`bool` 不能随手用，项目受 C89 约束。

- **能直接看到调试痕迹，而不只是最后改对了的结果。**  
  例如 `envdata.c` 里保留了很多这类标记：
  - `//DEBUG1: maxmin函数撞名`
  - `//errorchange: ...`
  - `//ed* i = &head;//errorchange`
  - `//i = i > -next;你踏马在写什么`
  
  这类内容的作用不是“美观”，而是把错误、误写、修正过程一起留在现场。读代码的人能看到问题是怎么发生的，而不只是看到修正后的写法。

- **能直接看到一些注释其实是在记录“为什么这样设计”。**  
  例如 `envdata.c` 里并不只是写“这里用了二级指针”，而是一路往下解释：
  - 什么时候传 `head`
  - 什么时候传 `&head`
  - 为什么函数里要永久修改调用者手里的指针
  - 为什么链表节点必须放在堆上而不是栈上
  
  同样，在搜索部分的注释里，也不是简单写“用了位掩码”，而是把放弃可变参数、转向结构体和位运算的原因都写出来了，包括“核心复杂度不会消失，只能组织和隔离复杂度”这种很具体的判断过程。

- **能直接看到一些很面向自己的、并不教科书化的提醒。**  
  例如：
  - `//malloc是会失败的孩子们`
  - `//但是id还是要int的~`
  - `//孩子，指针怎么可能滚呢`
  
  这类句子当然不属于正式文档语言，但它们确实记录了写代码时的思路状态：不是写给课程模板看的，而是写给当时那个还在反复出错、反复确认细节的自己看的。

- **能直接看到“重复工作交给 AI，但关键判断自己做”的痕迹。**  
  在 `src` 附近的注释里，甚至会直接写明：`//不同条件下的搜索逻辑、位运算、结构体输入和遍历这些“真算法”已经想清楚了，剩下只是重复劳动，所以才交给 AI 填`。这种写法比单纯在 README 里写一句“使用了 AI 辅助”更具体，因为它把边界写出来了。

- **能直接看到多文件工程和平台适配留下的工程痕迹。**  
  例如矩阵项目说明书里会明确写：
  
  - `main.c` 负责统一调度
  - `general.c` 负责初始化、稀疏化、CSR 转换
  - `basic.c`、`neon.c`、`kblas.c`、`huawei.c` 分别处理不同计算路径
  - 还专门设了 `local_mock.h`，只是为了让 VS 的 IntelliSense 和语法检查先工作起来
  
  这种内容和“算法多先进”关系不大，但能说明项目已经从单文件练习题变成了要处理头文件、模块划分、IDE 适配、Linux 编译路径的多文件工程。
  
- **能直接看到“服务器能跑”和“本地能写”之间是怎么折中的。**  
  例如矩阵项目里会单独写 Linux 编译适配，说明头文件路径不是默认就能找到，需要自己补 `-I`、`-L` 和运行时路径；同时又为了本地写代码方便，额外准备了假的头文件做语法辅助。这里保留的不是结果展示，而是环境折腾本身。
