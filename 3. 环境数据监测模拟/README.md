> ```c
> /* 更合理的归一化方式示意 */
> double best_log = logpost[1];
> int best_c = 1;
> for (c = 2; c <= 5; c++) {
>  if (logpost[c] > best_log) {
>      best_log = logpost[c];
>      best_c = c;
>  }
> }
> 
> double sum_exp = 0.0;
> for (c = 1; c <= 5; c++) {
>  sum_exp += exp(logpost[c] - best_log);
> }
> 
> double confidence = 100.0 / sum_exp;
> ```

## 1. 需求分析

根据课程给出的项目要求，本项目的基础目标是：使用 C 语言与链表实现一个城市环境监测数据分析与管理系统，完成环境监测数据的读取、添加、修改、删除、查询、排序以及文件读写，并在此基础上额外实现至少一个拓展功能。我最终选择的拓展方向是“空气污染等级预测”，即基于已有带标签数据实现朴素贝叶斯分类，而没有继续实现 K-means 聚类。

本项目需要解决的核心问题，是如何在控制台环境下对分散、多字段、可持续增删改查的环境监测数据进行组织和处理。围绕这一目标，我将项目需求归纳为以下几个部分：

- 数据管理：支持动态添加、删除、修改监测点数据。
- 数据查询：支持单条件或多条件组合查询，并返回符合条件的结果。
- 数据持久化：支持将链表中的数据保存为 CSV 文件，并从 CSV 文件中重新载入数据。
- 数据分析：支持按不同字段排序，并对无污染等级标签的数据进行污染等级预测。
- 用户交互：提供菜单式文本界面，尽量降低输入阻力，并给出明确反馈。

> 补充说明：课程给出的示例结构体字段并不算多，但我在正式实现时仍然保留并扩展到了 `位置`、`温度`、`湿度`、`PM2.5`、`风速`、`风向`、`日期`、`污染等级` 等字段。一方面，这是为了覆盖文件读写、搜索、排序与扩展算法的需要；另一方面，我在代码注释里也明确意识到：如果真的把所有污染物原始指标都完整引入，那么污染等级本身反而可以用标准公式直接计算，朴素贝叶斯就失去了这个作业语境下的意义。因此这里保留的是一个更贴近课程任务而不是贴近真实气象业务的数据设计。

## 2. 整体设计

### 2.1 功能设计

本项目整体采用“主控模块 + 链表核心模块 + 数据处理模块 + 文件读写模块”的组织方式。虽然最终代码仍集中在一个源文件中，但从逻辑职责上，我已经有了比较明确的模块划分。

- 主控模块：由 `main` 函数负责，承担菜单展示、用户输入分发和各功能入口调度。
- 链表核心模块：由 `ins`、`del`、`rls` 等函数组成，负责节点的创建、插入、删除和释放。
- 数据处理模块：由 `src`、`edi`、`ord`、`sta` 等函数组成，分别负责查询、修改、排序和污染等级预测。
- 文件读写模块：由 `sav`、`red` 两个函数组成，分别负责保存和载入 CSV 数据。
- 输入与提示模块：由 `get_integer_input`、`get_float_input`、`get_string_input`、`flg`、`tip`、`maintip` 等辅助函数组成，负责菜单按键、字段选择、输入读取和提示信息展示。

设计这一结构的目的，不是把代码写得像大型工业项目那样完全分文件，而是在课程作业范围内，尽可能让不同逻辑各司其职，避免把所有输入、计算、链表操作混在一个巨大流程里。

### 2.2 数据结构设计

本项目共设计了三个核心结构体。

#### 1. `ed`：主链表节点结构体

`ed` 是本项目最核心的数据结构，用于实际存储环境监测数据，并通过 `next` 指针串成单向链表。

```c
typedef struct envdata {
    int id;
    char loca[100];
    float temp;
    float humi;
    float pm25;
    float wdspd;
    int wddir;
    char date[12];
    int pollv;
    struct envdata* next;
} ed;
```

我选择链表而不是数组，主要出于以下考虑：

- 课程要求本身要求使用链表实现动态增删改。
- 本项目的数据条数在运行前并不固定，链表比定长数组更适合动态扩展。
- 删除和插入节点时，链表修改指针即可完成，不需要像数组那样整体挪动元素。

> 补充说明：关于 `head` 究竟该不该做成全局变量、为什么某些函数需要 `ed**` 而另一些只需要 `ed*`，这是我在调试过程中反复纠结的问题之一。最后代码采取的原则是：**需要修改头指针本身的函数，一律接收 `ed**`；只读遍历或基于已有头指针开展工作的函数，只接收 `ed*`。** 这个原则后来成为整个项目接口设计的基础。

#### 2. `eds`：查询条件结构体

`eds` 不代表真实监测数据，而是代表“查询请求” (search)。

```c
typedef struct envdatasrc {
    int idA;
    int idB;
    char loca[100];
    float tempA;
    float tempB;
    float humiA;
    float humiB;
    float pm25A;
    float pm25B;
    float wdspdA;
    float wdspdB;
    int wddir;
    char dateA[12];
    char dateB[12];
    int pollv;
} eds;
```

这样设计的原因在于，查询条件与真实节点的数据形态并不一致。例如：

- `ID` 查询需要上下界，而不是单个 `id`。
- `温度`、`湿度`、`PM2.5`、`风速` 等搜索通常需要区间。
- `位置` 查询使用关键词模糊匹配。
- `日期` 查询也更适合作为区间。

如果强行复用 `ed` 作为查询参数，就会出现语义混乱，也容易把“真实数据”和“搜索条件”混为一谈。因此我单独设计了 `eds`。

#### 3. `ede`：修改数据结构体

`ede` 对应“修改请求” (edit)。它只保留可编辑字段，不再包含 `id` 和 `next`。

```c
typedef struct envdataedi {
    char loca[100];
    float temp;
    float humi;
    float pm25;
    float wdspd;
    int wddir;
    char date[12];
    int pollv;
} ede;
```

我后来单独把它拆出来，而没有继续复用 `ed` 或 `eds`，主要是因为：

- 使用 `ed` 作为修改输入结构体，会混入 `id` 和 `next` 这类不该由用户直接填写的字段。
- 使用 `eds` 又会引入大量上下界字段，和“修改一个确定值”的语义不符。
- `ede` 作为最小化结构体，更适合作为修改操作的输入载体，也更便于配合位掩码按需更新字段。

### 2.3 算法设计整体思路

#### 1. 链表插入：头插法

本项目新增节点时采用头插法，即新节点直接接到链表头部。

这样选择的原因是：

- 实现简单，不需要遍历到尾部。
- 对链表来说，头插法时间复杂度更低，适合课程项目的基本 CRUD 场景。
- 当时我最需要优先解决的是“节点的正确创建、连接和释放”，而不是维持展示顺序；展示顺序的问题可以交给排序模块处理。

需要注意的是，`ins` 函数除了插入节点，还会遍历链表求当前最大 `id`，再将新节点编号设为 `max + 1`。这对手动添加数据是合理的，但在 `red` 函数中从文件载入已有数据时，这一做法就出现了轻度冗余：函数先生成一个新 `id`，随后又被文件中的原始 `id` 覆盖。这个设计当时是为了赶进度复用已有函数，放到现在看，应该拆分出“只分配节点不处理 ID”的更底层接口。

#### 2. 文件解析：`fgets` + `strtok`

文件读取最终采用的是按行读取，再按字段切分的方式。

```c
fgets(...) -> strtok(...) -> atoi / atof / strncpy
```

之所以没有继续使用 `fscanf` / `sscanf` 直接整行格式化读取，主要原因有三点：

- CSV 中包含字符串字段，直接格式化读取对空格、逗号、格式损坏更脆弱。
- 按字段切分后可以逐项判断是否出错，更便于进行“哪一行坏了就删掉哪一行”的处理。
- 逐字段处理也更方便配合长度检查、日期清洗和数值合法性校验。

> 补充说明：这一块的设计演进很明显。初版思路曾经试图直接把一整行按格式扫进去，但后来很快意识到 `%c` 只能读单字符、字符串字段不能这样处理、错误输入也会连锁污染后续字段，所以最终转向了 `fgets + strtok`。代码注释里还专门补记了 `strtok_r / strtok_s` 的线程安全问题，只是考虑到这是单线程课程项目，最后仍保留了 `strtok`。

#### 3. 多条件查询：位掩码 + 一票否决

多条件查询是本项目最重要的“结构设计题”之一。最后采用的是：

- 用一个 `unsigned int flag` 表示哪些字段被用户选中；
- 用 `eds` 保存这些字段对应的查询值；
- 在 `src` 函数中遍历链表，按 `flag` 逐项判断；
- 只要有任意一项条件不满足，就将当前节点判定为不匹配。

这一设计的优点在于：

- 接口比较稳定，不需要为不同数量的条件写不同函数。
- 位掩码使多条件组合的表达非常紧凑，适合 C 语言。
- 查询逻辑可以做到“按需判断”，没有被选中的字段不参与计算。

为了支持搜索结果后续排序和浏览，`src` 并不是直接打印命中的节点，而是把命中节点**复制**到一条新的链表 `head_src` 中，再将其返回。这样做虽然增加了一次内存分配，但它让“原始数据链表”和“搜索结果链表”分离开来，后续可以单独对搜索结果排序，而不影响总数据集。

#### 4. 排序：冒泡排序 + 交换节点数据

排序模块最终采用了冒泡排序，并通过 `swp` 函数交换节点中的字段数据，而不是直接交换链表节点本身。

这样做的理由很现实：

- 课程项目的数据规模有限，性能压力不大。
- 链表直接交换节点牵涉到前驱节点、头指针、`next` 重连等问题，出错概率高。
- 交换数据虽然会多写一些字段复制代码，但逻辑更直观，也更稳妥。

从现在的视角看，如果数据量上去，链表排序更适合使用归并排序；但在当时的项目时间和掌握程度下，选择冒泡排序是合理的。

#### 5. 拓展功能：朴素贝叶斯而不是 K-means

课程要求在 K-means 聚类和空气污染等级预测之间至少完成一项拓展功能。我最终实现的是朴素贝叶斯预测，而没有继续完成聚类。

> 我选择朴素贝叶斯的原因很现实。与 K-means 相比，朴素贝叶斯更容易在 C 语言里用数组和计数方式实现，尤其适合**时间很紧**的情况下落地...
>

由于朴素贝叶斯更适合离散特征，我又额外引入了“分箱”设计：

- 温度分为 6 档。
- 湿度分为 4 档。
- PM2.5 分为 6 档。
- 风速按**蒲福风级**思路分为 6 档。
- 风向分为 8 个方向。

这些分箱通过 `enum` 和 `get_xxx_bin` 系列函数实现，再借助三维数组 `ftr[5][8][6]` 统计不同类别下各特征箱体的频次，并通过拉普拉斯平滑避免零概率问题。

## 3. 关键模块详细设计

### 3.1 链表核心模块：`ins`、`del`、`rls`

#### `ins`

`ins` 函数负责：

- 在堆上申请一个新的 `ed` 节点；
- 扫描当前链表，找出最大 `id`；
- 为新节点分配新的 `id`；
- 采用头插法把新节点插入链表头部；
- 返回新节点地址，供后续字段填写使用。

该函数之所以返回 `ed*`，是因为我不仅需要“把节点插进去”，还需要拿到这个新节点的地址，继续向里面写入 `loca`、`temp`、`date` 等字段。

#### `del`

`del` 函数负责按 `id` 删除节点。其关键点在于：

- 删除头节点时，需要直接修改 `*head`；
- 删除中间节点时，需要同时维护“当前节点”和“前驱节点”；
- 删除完成后必须调用 `free` 释放内存。

这部分让我真正理解了“链表删除本质上是改连接关系，而不是简单清零”。

#### `rls`

`rls` 函数负责释放整个链表。它通过循环保存当前节点、移动到下一个节点、释放旧节点的方式，完成整条链表的清理，最后再把 `*head` 置为 `NULL`，避免悬挂指针。

### 3.2 文件读写模块：`sav`、`red`

#### `sav`

`sav` 函数的特点主要有三个：

- 使用 `GetSaveFileName` 弹出 Windows 保存对话框，让用户自主选择保存位置；
- 使用 `fprintf` 以 CSV 格式逐行写出链表数据；
- 针对 Excel 自动改写日期格式的问题，将日期保存为 `="YYYY-MM-DD"` 形式，以尽量保持文本原样。

#### `red`

`red` 函数是整个项目里实现最重的函数之一。它的流程大致如下：

1. 调用 `GetOpenFileName` 获取用户选择的 CSV 路径；
2. 打开文件，先读掉第一行表头；
3. 逐行使用 `fgets` 读取；
4. 对每一行先调用 `ins(head)` 预插入一个新节点；
5. 使用 `strtok` 按逗号切分字段；
6. 使用 `atoi`、`atof`、`strncpy` 分别写入各成员；
7. 用 `parse_error` 和 `qlf(newed)` 双重判断该行是否合法；
8. 如果不合法，就调用 `del(head, newed->id)` 删除这个脏节点；
9. 循环结束后用 `feof` / `ferror` 判断读取结束的原因。

这部分最大的价值不只是“能读文件”，而是逼着我真正处理了数据损坏、字段缺失、字符串过长、日期异常、Excel 兼容和输入溢出等现实问题。

### 3.3 查询模块：`src` + `flg` + `qlf_s`

查询模块分成三层：

- `flg`：负责通过 `_getch()` 和位掩码让用户选择查询字段；
- `qlf_s`：负责对 `eds` 中的区间关系进行总体验收；
- `src`：负责真正遍历链表并生成结果链表。

其中 `src` 的核心思想是：

- 遍历每个节点时，先假定它满足条件；
- 再按照 `flag` 对已激活字段逐一检查；
- 只要有任意一项不满足，就立刻判定为不匹配；
- 只有全部通过后，才把该节点复制到 `head_src` 中。

这种写法虽然看上去有一点“笨”，但对于课程项目非常清晰，也很便于后续定位具体字段出了什么问题。

### 3.4 修改与排序模块：`edi`、`ord`

#### `edi`

`edi` 函数不是通过让用户直接改节点，而是：

- 先由主流程找到要修改的目标节点；
- 再由 `ede` 保存一份“新数据”；
- 最后通过 `flag` 指示哪些字段需要覆盖。

这种设计让我把“找到谁”“改什么”“哪些字段有效”三件事分开了，后期读代码时更容易理解。

#### `ord`

`ord` 函数的思路是：

- 接收一个 `ed** target`，这样既可以排序主链表 `head`，也可以排序搜索结果链表 `head_src`；
- 通过单个 `flag` 指定排序依据；
- 在双层循环中比较相邻逻辑顺序是否正确；
- 若不正确，则调用 `swp` 交换两个节点的数据内容。

这部分让我第一次比较清楚地理解了“排序对象可以是链表本身，也可以是链表的某个副本结果集”这一层抽象。

### 3.5 扩展功能：朴素贝叶斯预测模块 `sta`

`sta` 函数负责对所有 `pollv == 0` 的节点进行污染等级预测。其主要步骤为：

1. 遍历主链表，统计所有已知污染等级节点的类别数量 `cat[c]`；
2. 按特征分箱，对每个类别的各箱体计数 `ftr[f][b][c]`；
3. 根据类别计数计算先验概率 `pri[c]`；
4. 根据箱体计数计算条件概率 `lik[f][b][c]`，并使用拉普拉斯平滑；
5. 对待预测节点取出各特征的箱体编号；
6. 计算各类别的对数后验值 `logpost[c]`；
7. 取最大者作为预测结果，并输出一个相对意义上的“置信度”。

这里我采用对数形式的原因很直接：

- 五个条件概率连续相乘后数值会极小；
- 直接使用普通乘法会更容易出现下溢；
- 转成 `log` 以后，可以把连乘转换成加法，计算更稳定。

不过正如归档说明里已经指出的，这一部分在最终提交时仍然有两个明显局限：

- 显示给用户的“置信度”并不是真正严格意义上的后验概率，只是一个归一化后的相对分数；
- 而我当年在归一化实现上又少算了 `c = 1` 类别，导致某些情况下百分比超过 `100%`。

## 4. 编码与实现说明

### 4.1 开发环境

- 操作系统：Windows
- 编译环境：Visual Studio 2022 / Dev-C++ 5.11
- 编程语言：C
- 兼容目标：尽量按 C89 / ANSI C 风格编写，同时使用部分 Windows 特有 API

### 4.2 用户交互实现

本项目没有采用复杂图形界面，而是使用纯文本菜单进行交互。

菜单实现上，我最后没有继续使用 `scanf` 读取主菜单选项，而是改用了 `_getch()`，主要因为：

- `_getch()` 可以立即响应，无需用户再按回车；
- 它不会把换行符残留在缓冲区里，从而减少后续输入被污染的问题；
- 对控制台小项目而言，交互手感会更直接。

围绕这一点，代码中还专门实现了：

- `maintip()`：主菜单展示；
- `tip()`：针对不同字段动态展示输入说明；
- `flg()`：基于 `_getch()` 和位掩码实现单选 / 多选字段切换。

### 4.3 输入校验实现

为了减少无效输入造成的连锁问题，我没有把所有输入都直接交给 `scanf`，而是拆成三个“活字函数”：

- `get_integer_input`
- `get_float_input`
- `get_string_input`

它们统一基于 `fgets` 读取一整行，再做轻量解析。这样可以较好地避免：

- `scanf` 遇到错误输入后卡住缓冲区；
- 用户直接回车时与“无输入”状态混淆；
- 输入后半段残留影响下一次读取。

### 4.4 提示语与输出风格

本项目的控制台提示语没有做成完全严肃的风格，而是故意保留了一些更轻松的表达，例如：

- 搜索成功后的提示；
- 删除、保存、读取、取消时的反馈；
- 数据输入不合法时的提醒。

这样做一方面是为了降低长时间调试时的枯燥感，另一方面也让一个纯控制台项目更有一点可读性和“反馈感”。

## 5. 测试与调试记录

这一部分是本项目最有价值的内容之一。因为真正把这个项目做出来以后，我发现自己反复踩中的并不是某一个单点错误，而是几类会不断循环出现的问题。

### 5.1 典型功能测试

#### 1. 文件读取与展示测试

- 准备一个包含多条记录的 CSV 文件。
- 通过菜单选择“从文件加载数据”，使用文件对话框读取数据。
- 再选择“显示所有数据”，检查每条记录是否能被完整展示。

预期结果是：

- 表头被跳过；
- 每一行合法数据都能被读入链表；
- 非法行会被识别并跳过；
- 所有合法节点都能按当前链表顺序正确显示。

#### 2. 多条件查询与排序测试

- 先读取一份包含不同位置、温度、日期和污染等级的 CSV。
- 在查询模块中同时勾选 `位置`、`温度`、`日期` 等字段；
- 输入相应关键词与范围；
- 得到搜索结果后，再对搜索结果链表执行排序。

预期结果是：

- 只有同时满足所有条件的记录会进入结果链表；
- 原始总链表不被破坏；
- 搜索结果可以单独排序和显示。

#### 3. 污染等级预测测试

- 保证当前数据集中同时存在 `pollv != 0` 的已知样本和 `pollv == 0` 的未知样本；
- 进入“数据统计分析”功能；
- 观察程序是否能够识别未知样本并输出预测等级。

预期结果是：

- 程序能够先用已知样本完成计数和概率构建；
- 再对未知样本输出预测等级；
- 结果能正常显示，但“置信度”存在前文已说明的归一化缺陷。

### 5.2 反复出现的主要问题

#### 1. `head` 到底该传几层指针

这是我最反复纠缠的问题之一。表现包括：

- 明明想改链表头，却只传了 `head`；
- 明明只是遍历，却又错误地传成了 `&head`；
- 把 `*head`、`&head`、`ed*`、`ed**` 混着用；
- 在某些地方能编译，在另一些地方又完全不对。

这个问题最终逼着我把“修改头指针”和“只读取头指针”彻底区分开来。也正是在这之后，整个项目的函数接口才真正稳定下来。

#### 2. 数组、结构体、字符串复制规则混淆

这一类问题也出现得非常频繁，典型表现包括：

- 误以为数组可以像结构体那样直接赋值；
- 误用 `strcpy` 导致长度风险；
- 后来又补成 `strncpy`，并手动补 `\0`；
- 对日期字段长度估计不足，导致回车残留在缓冲区里。

这些问题看起来琐碎，但实际上一旦处理不好，就会直接把输入、文件读取和输出全部带歪。

#### 3. 输入函数与缓冲区残留

这也是我反复出问题的一块，而且它麻烦的地方不在于“不会写输入函数”，而在于 **不同输入函数各自对应不同的输入流行为**。项目早期我用过 `scanf`，很快就发现它在面对非法输入时容易卡住：比如本来要读整数，用户却输入了非数字字符，这些字符会继续留在 `stdin` 里，下一轮读取又会再次失败，程序就会表现出近似“死循环”的状态。后来我改用 `getchar` 和 `fgets` 试图绕开这个问题，但又碰到了另一个更隐蔽的麻烦：如果某次读取没有把整行吃干净，残留在缓冲区里的 `\n` 会被下一次 `fgets` 立刻读走，结果表面上看像是“函数没等我输入就直接返回了”。调试记录里把它形容成“幽灵换行符”，这个说法其实很准确。

这个问题之所以在本项目里特别明显，是因为我的输入场景很多，而且它们混在一起：主菜单选择、字段勾选、日期输入、查询条件输入、精确匹配和区间匹配的切换，并不是都适合同一种方式。后来主菜单和字段选择改成了 `_getch()`，它的好处是直接读按键、不经过标准输入缓冲区，因此不会留下额外的换行残留；但与此同时，具体数据录入又仍然需要字符串和数值解析，于是我最终形成了分层方案：**菜单类、单按键类输入用 `_getch()`；真正需要读取一整行文本的地方用 `fgets`；再在其上叠 `sscanf` 或手动解析。** 这样做的本质，是把“按键选择”和“文本输入”分开处理，而不是继续让它们共用一套 `scanf` 逻辑。调试记录中也明确提到，`main` 和 `flg()` 用 `_getch()` 获取选择，而数据输入部分再交给封装后的 `get_integer_input`、`get_float_input`、`get_string_input`，目的就是尽量减少不同输入机制互相污染的机会。

再往后一步，我才真正意识到这里不只是“换一个函数”这么简单，而是需要把输入处理做成**独立组件**。也就是说，我后来不再把“读取一个整数”“读取一个浮点数”“读取一个字符串”直接散写在业务逻辑里，而是把它们封装成统一的小函数，让每次读取都尽量走“先读整行，再判断是否合法”的路线。这样做虽然比直接 `scanf` 多了一层，但好处非常直接：出错时更容易定位，缓冲区行为更可控，也更适合和日期、范围、关键词这些不同输入语义配合。这个演进过程在调试实录里概括得很清楚：方案是从 `scanf`，一路折腾到 `getchar`，再到 `fgets + sscanf`，最后再把菜单部分抽离给 `_getch()`。对这个项目而言，输入函数的变化其实就是一次典型的“从能读数据，走到能稳定处理错误输入”的过程。

#### 4. 文件读写与外部软件兼容

这个问题来源并不是 C 语言本身，而是 **CSV 作为纯文本格式** 与 **Excel 作为“会主动猜类型的软件”** 之间的冲突。课程任务只要求“把链表数据写入文件、再从文件读回”，表面上看只是普通的 `sav` / `red` 模块；但真正做起来我才发现，程序写出的 CSV 并不会只被我的程序读取，它还会被 Excel 这种外部工具打开，而一旦经过它，文件内容就可能不再保持原样。调试记录中，这一块后来被明确当作一次“外部依赖处理”的拉锯战来看待。

这里最麻烦的字段是日期。对我的程序来说，日期本来只是一个固定格式的字符串，例如 `2025-11-19`；但 Excel 在打开 CSV 时，并不会把它老老实实当作纯文本，而是会先猜测“这像不像日期”，然后按它自己的规则重写显示和保存格式。这里的“规则”并不是我程序里的规则，而是 Excel 的自动类型推断规则：它可能把 `2025-11-19` 识别成日期单元格，**再显示成 `2025/11/19`、`2025/11/9`，甚至去掉前导零**；如果用户再保存一次，这种改写后的内容就会真正写回 CSV。这样一来，程序原本保存的是标准的 `YYYY-MM-DD` 文本，但文件经过 Excel 打开和保存之后，磁盘上的实际字符串已经变了。

这个问题之所以重要，是因为它会直接反过来破坏我自己的读取逻辑。`red` 函数在设计时默认文件中的日期字段应当符合既定格式，再配合 `fgets + strtok` 做逐字段切分和校验；如果日期被 Excel 改成了另一种写法，那么后续的长度检查、字符串比较、日期区间判断，都会受到影响。也就是说，这已经不是“显示上有点不好看”的问题，而是“同一个程序自己写出去的数据，可能再也按原规则读不回来”的问题。课程模板里只说要实现文件读写，但真正落地时，我遇到的其实是**持久化格式与外部软件行为之间的兼容性问题**。

围绕这个问题，我当时尝试过几种思路。一个方向是增强读取端的兼容能力，也就是让 `red` 接受更多日期写法；另一个方向是从保存端入手，尽量让 Excel 在打开时就不要乱改。调试记录里先讨论过在保存时给字段加 `\t` 的折中方案，它的思路是“迷惑 Excel”，让 Excel 倾向于把该字段看成文本；但这并不彻底，因为它主要只能缓解“打开时自动格式化”，无法保证用户再次手动编辑和保存后不出问题，而且不同软件对这种写法的兼容性也不完全一致。

在这个基础上，后来又进一步采用了更强的文本伪装思路，也就是把日期字段在保存时包装成 `="2025-11-19"` 这种形式。这样做的本质，不是为了让文件“更漂亮”，而是为了**强制 Excel 把这一列当作文本结果而不是日期值**。调试记录里把它称作“用等号和引号欺骗 Excel”，因为对 Excel 来说，这看起来像一个公式，而公式结果又是一个字符串，于是它就不会再按日期格式重写这个字段。与此同时，读取端再补一层“净化”或“剥离包装”的处理，把 `="..."` 外壳去掉，还原为程序内部真正需要的 `YYYY-MM-DD` 字符串。

这一点的价值在于，它让我第一次比较清楚地意识到：所谓“文件格式设计”并不只是决定字段顺序、分隔符和结构体映射，还包括**预判文件会被谁打开、被谁改写、被谁再保存**。CSV 本身只是纯文本，但只要现实中默认会有人用 Excel 去碰它，那么程序就必须面对 Excel 的自动类型推断。换句话说，我这里处理的已经不是单纯的 C 语言字符串问题，而是一个很典型的工程兼容性问题：程序内部的数据格式设计，必须考虑外部软件的行为模型，否则“保存成功”并不等于“数据可稳定往返”。这一段经历虽然看起来只是日期字段的小修补，但它实际上让我第一次接触到了很真实的数据清洗思路，也就是：**接收外部数据、识别外部改写、再把它标准化为程序内部可控格式**。

### 5.3 主要 debug 演进

结合源码中的 `DEBUG`、`errorchange`、`advancechange` 标记，以及调试实录，可以把本项目的 debug 过程概括为以下几条主线：

1. `DEBUG1`：函数名冲突问题，例如 `max` / `min` 的命名冲突，后来改成 `maxi` / `mini`。
2. `DEBUG2`：结构体定义中的基础语法问题，例如字段后的分号缺失。
3. `DEBUG3`：`typedef` 别名理解不清，导致 `envdata**` 与 `ed**` 混用。
4. `DEBUG4` / `DEBUG5`：变量名写错，例如 `dir` / `wddir`、`i` / `current` 等复制性错误。
5. `DEBUG7`：意识到 C89 没有标准 `bool`，需要用 `int` 或自行定义兼容写法。
6. `DEBUG8`：文件读取时的行号变量命名不一致。
7. `DEBUG9` 到 `DEBUG13`：围绕 `head`、`*head`、`&head` 的函数调用反复修正，以及对 `head_src`、排序目标链表等关系的逐步理顺。
8. `errorchange`：通常表示“已经定位出明确错误并直接修补”的修改，例如参数顺序、输入函数类型不匹配、日期数组大小等。
9. `advancechange`：通常表示“程序已经能跑，但还需要进一步增强健壮性或兼容性”的修改，例如 CSV 日期兼容、清空链表时先保存下一节点、输入为空时的静默处理等。

> 补充说明：简直是开发现场记录，非常便于复盘和警醒。很多被注释掉的旧代码都没有直接删掉，而是被保留成了“问题尸体”和修复路径，它们的价值不比最终留下来的正确代码低。

### 5.4 朴素贝叶斯模块的剩余问题

这个模块虽然已经实现，但在最终提交时仍然带着明显的赶工痕迹：

- 置信度归一化有错误，可能超过 `100%`。
- `sta` 命名与“统计分析”不完全一致，功能已经偏向预测。
- 没有额外做模型评估，例如准确率、混淆矩阵等。
- 分箱规则主要以“够用”和“便于编码”为目标，并没有系统调参。

换句话说，这个扩展功能在课程语境下已经完成了“从零实现一个可运行的朴素贝叶斯预测流程”的任务，但如果从更严格的算法实验角度看，它还远远不是一个完成度很高的模型实现。

## 6. 总结与反思

### 6.1 已完成内容

本项目最终完成了以下目标：

- 使用链表实现环境监测数据的动态存储。
- 实现了添加、显示、查询、修改、删除、排序等基础数据管理功能。
- 实现了 CSV 文件的保存与读取。
- 实现了带位掩码的多条件组合查询。
- 实现了朴素贝叶斯污染等级预测这一拓展功能。
- 在控制台环境下做出了一套相对完整的菜单与提示系统。

### 6.2 不足与需要改进的地方

从现在来看，本项目仍有以下改进空间：

- 将不同模块拆分为独立 `.h` / `.c` 文件，降低耦合度。
- 将排序从冒泡排序替换为更适合链表的归并排序。
- 将 `strtok` 替换为更安全的版本，进一步规范字符串解析。
- 对文件读取中的数值合法性做更严谨的解析，而不是主要依赖首字符和长度判断。
- 为朴素贝叶斯模块增加更完整的概率归一化与测试评估。
- 重新审视 `head_src` 的所有权与使用范围，减少结果链表复制带来的复杂度。
- 处理更多菜单取消、查询为空、节点不存在等边界情况。
